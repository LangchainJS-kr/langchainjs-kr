# 보안

LangChain은 로컬 및 원격 파일 시스템, API 및 데이터베이스와 같은 다양한 외부 리소스와의 통합을 포함하는 큰 생태계를 가지고 있습니다. 이러한 통합을 통해 개발자는 LLM의 강력한 기능과 외부 리소스에 접근, 상호작용하면서 조작할 수 있는 다재다능한 애플리케이션을 만들 수 있습니다.

## 모범 사례

이러한 애플리케이션을 구축할 때 개발자는 올바른 보안 관행을 준수해야 합니다.

- [**최소 권한의 원칙**](https://ko.wikipedia.org/wiki/%EC%B5%9C%EC%86%8C_%EA%B6%8C%ED%95%9C%EC%9D%98_%EC%9B%90%EC%B9%99): 권한을 애플리케이션의 니즈에 맞게 구체적으로 설정하세요. 광범위하거나 과도한 권한을 부여하면 심각한 보안 취약점이 발생할 수 있습니다. 이러한 취약점을 피하기 위해 읽기 전용 자격 증명 도입하거나 민감한 리소스에 대한 접근을 차단하고, 애플리케이션에 적합한 샌드박싱 기술(예: 컨테이너 내부에서 실행)을 사용하는 것을 고려하세요.
- **잠재적 오용 예상**: 인간이 실수를 할 수 있는 것처럼 거대 언어 모델(LLM)도 실수를 할 수 있습니다. 시스템 접근 권한이나 자격 증명이 부여된 권한 내에서 어떠한 방식으로든 사용될 수 있다는것을 인지해야 합니다. 예를 들어, 데이터 삭제를 허용하는 데이터베이스 자격 증명이 있다면, 해당 자격 증명을 사용할 수 있는 LLM이 실제로 데이터를 삭제할 수 있다고 가정하는 것이 가장 안전합니다.
- [**심층 방어**](<https://ko.wikipedia.org/wiki/%EB%A0%88%EC%9D%B4%EC%96%B4%EB%93%9C_%EC%8B%9C%ED%81%90%EB%A6%AC%ED%8B%B0>): 완벽한 보안 기술은 없습니다. 파인튜닝 및 좋은 체인 설계를 통해 거대 언어 모델(LLM)이 실수할 확률을 줄일 수 있지만 완전히 제거할 수는 없습니다. 단일 방어층에 의존하기보다는 여러 계층의 보안 접근 방식을 결합하는 것이 가장 좋습니다. 예를 들어, 읽기 전용 권한과 샌드박싱을 모두 사용하여 LLM이 명시적으로 사용하도록 허가된 데이터만 접근할 수 있도록 하세요.

이를 하지 않는다면 아래와 같은 문제점이 있을 수 있고, 이 목록들이 전부는 아닙니다:

- 데이터 손상 또는 손실.
- 기밀 정보에 대한 무단 접근.
- 중요한 리소스의 성능 또는 가용성 저하.

예방 전략을 포함한 예시 시나리오:

- 사용자가 파일 시스템에 접근할 수 있는 에이전트에게 삭제해서는 안 되는 파일을 삭제하거나 민감한 정보를 포함하는 파일의 내용을 읽도록 요청할 수 있습니다. 이를 예방하기 위해 에이전트가 특정 디렉토리만 사용하도록 제한하고, 안전한 파일만 읽거나 쓰도록 허용하세요. 추가적으로 에이전트를 컨테이너에서 실행하여 샌드박싱하는 것을 고려하세요.
- 사용자가 외부 API에 쓰기 권한이 있는 에이전트에게 악의적인 데이터를 API에 쓰거나 API에서 데이터를 삭제하도록 요청할 수 있습니다. 이를 예방하기 위해 에이전트에게 읽기 전용 API 키를 부여하거나 이러한 오용에 이미 대응된 엔드포인트만 사용하도록 제한하세요.
- 사용자가 데이터베이스에 접근할 수 있는 에이전트에게 테이블을 삭제하거나 스키마를 변경하도록 요청할 수 있습니다. 이를 예방하기 위해 자격 증명을 에이전트가 접근해야 하는 테이블로만 제한하고 읽기 전용 자격 증명을 발급하는 것을 고려하세요.

파일 시스템, API 또는 데이터베이스와 같은 외부 리소스에 접근하는 애플리케이션을 구축하는 경우, 보안 팀과 상의하여 애플리케이션을 가장 잘 설계하고 보호하는 방법을 결정하는 것이 좋습니다.
